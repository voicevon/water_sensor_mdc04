## 1. 协议设计背景

为了在污水采样系统中支持多点分布式测量，同时最大限度地兼容现有中控网关（`water_relay`）和移动端 App（`water_android`）的解析逻辑，避免修改下游系统的繁琐工作，传感器节点进行了如下双模式重构：

* **I2C 模式（单芯片 - 4 传感器）**：用于小范围部署，仅挂载 1 个 MDC04，输出该芯片的 4 路传感器电容数据。
* **单总线/One-Wire 模式（三芯片 - 4 传感器/3路有效）**：用于多点部署，采用 3 条独立的 One-Wire 总线分别连接 3 个 MDC04 芯片（分别对应 GPIO 13、14、27）。
  * **传感器精简设计**：每颗芯片仅启用 1 路通道进行测量，另外三路通道关闭。
  * **协议复用**：将 3 颗芯片采集到的电容数据分别映射为 `sensor1`、`sensor2`、`sensor3`，第四路 `sensor4` 固定输出 `0` 作为预留占位。
  * **向下兼容**：该设计成功将 3 芯片的数据汇总为标准的 4 传感器结构，从而完全复用了原本的 4 通道（传感器）传输协议，无需任何协议包扩展。

## 2. BLE 蓝牙广播协议规范

为保持与旧版系统的无缝兼容，BLE 广播参数和数据包格式完全恢复到 V1.0 的 4 传感器标准。

### 2.1 基础广播参数
* **设备广播名称**：保持为 **`"FengBLE"`**（F大写，eng小写，BLE大写）。网关与 Android 端的过滤规则无需修改。
* **Company ID**：保持为 `0xFFFF`。
* **广播报文形式**：非连接的非定向广播（`ADV_NONCONN_IND`）。

### 2.2 广播数据（ADV_IND）字节布局 (15 字节)

主广告包均统一采用以下 15 字节的数据帧格式（Manufacturer Data 扩展为 12 字节）：

```
+--------------+---------------+--------------------------------------------------------------+
| AD Type Name | AD Type Mfg   |                         LTV 数据                             |
|   (8 字节)   |   (6 字节)    |                                                              |
+--------------+---------------+---------------+---------------+-------+------+-------+-------+
|  "FengBLE"   |  Company ID   |    sensor1    |    sensor2    |sensor3|sensor4|stateB| seqNum|
|  (6 字节+H)  | (2 字节, 0xFF) | (2 字节, MSB) | (2 字节, MSB) |  ...  |  ...  | (1 B) | (1 字节)
+--------------+---------------+---------------+---------------+-------+------+-------+-------+
```
* **电容数据映射**：大端序（MSB 在前，LSB 在后），物理电容值 = `value / 100.0` (保留两位小数)。
* **状态指示字 (stateB / State Byte)**：位于电容数据与 seqNum 之间。
  * **高 4 位**：恒为 `0` (`0b0000`)。
  * **低 4 位**：对应 4 路传感器的开关水状态：
    - `Bit 0` (LSB)：Sensor 1 状态 (`1` 有水, `0` 无水)
    - `Bit 1`：Sensor 2 状态
    - `Bit 2`：Sensor 3 状态
    - `Bit 3`：Sensor 4 状态
* **单总线模式映射关系**：
  - `sensor1` $\rightarrow$ 由配置网页映射选择的物理通道数据与有无水状态。
  - `sensor2` $\rightarrow$ 映射通道 2 数据与状态。
  - `sensor3` $\rightarrow$ 映射通道 3 数据与状态。
  - `sensor4` $\rightarrow$ 映射通道 4 数据与状态（默认占位）。

## 3. MQTT 通信协议规范

### 3.1 基础信息
* **启动/停止控制主题**：`water/sensor/start` (QoS = 2)。
* **数据上报主题**：`water/sensor/status` (QoS = 1)。
* **设备唯一标识（name）**：`"dongzhan"`。

### 3.2 JSON 载荷规范
MQTT 统一采用 4 传感器 JSON 包发送，格式如下：

```json
{
  "name": "dongzhan",
  "sensor1": 1500,
  "sensor2": 1505,
  "sensor3": 1495,
  "sensor4": 0
}
```
* **I2C 模式**：`sensor1`~`sensor4` 对应单芯片的 4 路通道。
* **单总线模式**：`sensor1` $\rightarrow$ Chip 1 Channel 1, `sensor2` $\rightarrow$ Chip 2 Channel 1, `sensor3` $\rightarrow$ Chip 3 Channel 1, `sensor4` $\rightarrow$ `0`。

## 4. 下游系统兼容性说明

由于通信协议（BLE 广播格式、广播名称 `"FengBLE"` 以及 MQTT 载荷字段）完全维持 4 传感器标准：
* **中控网关 (`water_relay`)**：无需进行任何代码修改或协议适配，能直接解析并处理广播和 MQTT 数据。
* **移动端 App (`water_android`)**：无需升级至 9 传感器图表，界面能够直接显示从 3 颗芯片采集到的 3 个传感器数据。
